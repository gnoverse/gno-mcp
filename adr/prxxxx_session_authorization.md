# Session Authorization

## Context

`gno-mcp` v1 signs transactions with the user's master key directly. The master key is loaded into the MCP process and used by write tools (`gno_call`, `gno_run`, etc.). Any compromise of the MCP â€” a bug, a supply-chain attack on a Go dependency, a host-level intrusion â€” exposes the full master signing capability.

`gnolang/gno` PR #5307 (`feat(auth): account sessions`, merged 2026-04-21) introduced chain-enforced delegated signing: an `auth.MsgCreateSession` registers a session pubkey bound to a master account, with declared `AllowPaths`, `SpendLimit`, `SpendPeriod`, and `ExpiresAt`. Every session-signed transaction is validated against the scope by the chain.

Inspection of the merged code:

- `auth.MsgCreateSession.SessionKey` has type `crypto.PubKey` â€” public material only; no private key in the message.
- `MsgCreateSession.GetSigners()` returns only the master account address; the session key does not sign this tx.
- The CLI surface (`gnokey maketx session create --pubkey <bech32> <master-key>`, added in PR #5614) accepts the session pubkey as a bech32 string flag, parsed via `crypto.PubKeyFromBech32`. The session privkey is not required to exist in the master's keyring.

The session keypair can therefore be generated by any party and authorized later by the master signing `MsgCreateSession` independently. The session privkey never has to touch the master's side.

## Decision

gno-mcp v2 signs writes exclusively with chain-bounded session keys. The master key is never accessible to the MCP process.

**Authorization flow:**

1. Agent invokes a write tool.
2. gnomcp checks for an active session matching the target profile and scope. If absent, returns `isError: true` with `code: authentication_required`.
3. Agent calls `gno_session_propose(profile, allow_paths[], spend_limit?, expires_in?)`.
4. gnomcp generates an ed25519 session keypair in process memory and emits a payload containing the proposed scope and the exact `gnokey maketx session create --pubkey <bech32> --allow-paths ... <master>` command for the user to run on their own machine.
5. User reviews the proposed scope, runs the command via their own `gnokey` (or wallet), signs with their master, broadcasts `MsgCreateSession`.
6. Agent retries the original write tool. gnomcp performs an on-demand chain query, finds the session active, signs the broadcast with the session privkey.

Sessions are immutable after creation; scope expansion requires a new `MsgCreateSession`. Multiple sessions per profile are supported; gnomcp picks the most recently created session whose `AllowPaths` matches the requested realm and that has not expired or run down its `SpendLimit`. Session private keys live in process memory only; restart re-prompts authorization.

**Scope policy** resolves `gno_session_propose` arguments through four layers (priority high to low):

1. Agent-explicit values.
2. Per-profile overrides in `profiles.toml` (`default-spend-limit`, `default-expires-in`).
3. Hardcoded hard defaults: `spend_limit = 100000ugnot`, `expires_in = 1h`. No default for `allow_paths` â€” agent must supply.
4. Hardcoded hard-limit clamps per chain-type:

| chain-type | max `spend_limit` | max `expires_in` | max `allow_paths` count |
|---|---|---|---|
| `local` | 100gnot | 30d | 20 |
| `testnet` | 10gnot | 7d | 10 |
| `mainnet` | 1000ugnot (placeholder; needs calibration against real gas costs) | 1h | 3 |

Any value resolved through layers 1â€“3 that exceeds the layer-4 limit is clamped to the limit and a warning is included in the tool result text so the user sees the clamp.

A per-profile `bypass-hard-limits = true` escape hatch disables layer 4 for that profile. Even with the bypass, mainnet writes still require `allow-dangerous-tools = true` separately.

## Alternatives considered

**Continue v1's master-key signing.** Rejected. Closes the Rule of Two trifecta (state-changing tools + untrusted input + sensitive access). The meta-issue (`.mynote/gno-agentic/issues/meta-issue.md`) defers open-ended writes specifically because this posture is unacceptable.

**gno-mcp PR #3's fund-to-authorize model.** Partially adopted: the OAuth-style UX flow (lazy keypair, `authentication_required` error with payload, user authorizes out of band) is taken. The chain-side mechanism is upgraded from "session is its own account, user funds it" to "session is a chain-bounded `MsgCreateSession` registration, user signs scope at authorization time." PR #3 itself notes the upgrade as future work. The funded-balance bound is replaced by chain-enforced `AllowPaths` + `SpendLimit` + `ExpiresAt`.

**Active-profile state machine** with a `select_profile` tool that mutates server state. Rejected. Adds session-scope state-management complexity without safety gain over per-call hard errors plus capability gates. State transitions add ambiguity to the audit log; stateless `profile`-arg on every call is cleaner.

**Per-profile manual scope without hardcoded floors.** Rejected. Most users will not edit `profiles.toml`; sensible safe defaults and a chain-type-aware ceiling must be structural, not opt-in.

**Persistent on-disk session keys.** Deferred. v2 holds session privkeys in process memory only. Restart re-prompts. Encrypted-on-disk persistence will be reconsidered if 24h re-authorization friction proves significant in practice.

## Consequences

- Master-key exposure is closed structurally. Risk axis A (per the design notes in `.mynote/v2-brainstorm/`) is eliminated.
- Blast radius (risk axis B) is bounded by chain-enforced session scope. Damage from a confused or compromised session is capped by `SpendLimit`, restricted to realms in `AllowPaths`, and expires automatically.
- Open-ended invocation (risk axis C) is **not** addressed by sessions. A session-authorized agent can still call `realm.AnyFunc(weird_args)` for any realm in `AllowPaths`. This is the shape of `(realm, func, args)` tools. The decision to keep these tools opt-in dev-mode (per separate ADR on tool surface) addresses C; a2a and x402 (future ADRs) address it for agent-driven flows.
- v1's master-key code paths (signer arg on tool calls, `gnokey` keyring integration) are removed wholesale. No backward-compatibility path.
- v1's per-call `confirm=true` mainnet gate is dropped. Chain-enforced session scope is stronger: the user reviewed the realm + spend cap + expiry at sign time, before any individual broadcast.
- Audit log entries (still written to disk by gnomcp) record the `session_address` used to sign each write.
- The MCP must perform on-demand chain queries (no background polling) to detect session activation and revocation. State transitions happen inside tool calls.
- Mainnet hard-limit calibration (1000ugnot / 1h placeholder) requires empirical follow-up. Expected to be tuned against real `MsgAddPackage` and small-call gas costs once mainnet is actively used.
