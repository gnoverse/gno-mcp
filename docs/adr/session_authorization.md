# Write Authorization: Agent Identity + Chain-Bound Sessions

**Status: implemented. The session path is declared WIP in user-facing docs (functional end-to-end, expected to be reworked).**

## Context

The v1 server signed transactions with the user's master key loaded into the MCP process. Any compromise of the MCP — a bug, a supply-chain attack, a host-level intrusion — exposed the full master signing capability.

`gnolang/gno` PR #5307 (`feat(auth): account sessions`) introduced chain-enforced delegated signing: `auth.MsgCreateSession` registers a session pubkey bound to a master account with declared `AllowPaths`, `SpendLimit`, and `ExpiresAt`; every session-signed transaction is validated against that scope by the chain's ante handler. The session keypair can be generated by any party and authorized later by the master signing `MsgCreateSession` — the session privkey never touches the master's side, and the CLI (`gnokey maketx session create --pubkey <bech32> ...`) accepts the pubkey as a string.

Only dev/testnet chains are reachable (chain-id allowlist, separate ADR), which changes the risk calculus for an MCP-owned key: testnet funds are the only thing at stake.

## Decision

Writes are signed by one of two identities, selected per call via the `identity` argument. The user's master key never enters the gnomcp process in either path.

**Agent identity (default).** gnomcp signs with its own key:

- **Local (`dev`) profiles** use the well-known public `test1` account — pre-funded by gnodev, zero setup.
- **Testnet profiles** use a per-profile key generated by `gno_key_generate` (BIP39, secp256k1), persisted under `~/.local/share/gnomcp/agent-keys` (mode `0600`; encrypted at rest with scrypt+AES-256-GCM when `GNOMCP_SESSION_PASSPHRASE` is set). Funding goes through `gno_faucet_fund` (tiered: faucet service → faucet page link → manual instructions). An unfunded write fails fast with a structured `insufficient_funds` error pointing at the faucet tool.

**Session identity (`identity=session`).** The agent acts *as the user* on profiles that declare a `master-address`:

1. `gno_session_propose(profile, allow_paths[], allow_run?, spend_limit?, expires_in?)` generates an ed25519 session keypair locally and emits the exact `gnokey maketx session create` command (`--allow-paths vm/exec:<realm>` per realm, `vm/run` when `allow_run`, `--spend-limit`, `--expires-at`).
2. The user reviews the scope and runs the command with their own `gnokey`; their master signs `MsgCreateSession` out of band.
3. On the next session write or `gno_auth_status` call, gnomcp queries `auth/accounts/<master>/session/<sessionAddr>` on demand, promotes the pending session to active, and signs the broadcast with the session privkey (`MsgCall.Caller = master`, `Signature.SessionAddr = session`).
4. `gno_session_revoke` emits the matching `gnokey maketx session revoke` command; revocation is user-signed like creation.

Sessions are immutable after creation; scope expansion means a new session. Among multiple matching sessions, gnomcp picks the most recently created active one whose scope covers the call.

**Scope policy** resolves proposal arguments through four layers (high to low): agent-explicit values → per-profile defaults (`default-spend-limit`, `default-expires-in`) → hardcoded defaults (`100000000ugnot`, sized so a default session covers ~10 calls at the chain's 10M-ugnot per-tx GasFee reservation; `1h`) → hard-limit clamps derived from chain-id locality:

| locality | max `spend_limit` | max `expires_in` | max `allow_paths` |
|---|---|---|---|
| local (`dev`) | 100gnot | 30d | 20 |
| testnet (`test*`) | 100gnot | 7d | 10 |

Values exceeding a clamp are clamped with a warning in the tool result. `bypass-hard-limits = true` disables the clamp layer for a profile. `allow_paths` has no default — the agent must supply it (or `allow_run=true`).

**Spend tracking** deducts the full tx `GasFee` (not `GasUsed`) from the local spend balance, matching the chain's session accounting, and persists it.

**Persistence and reconciliation.** Session keys live one file per session at `~/.local/share/gnomcp/sessions/<profile>/<session_addr>.key` (mode `0600`; same opt-in encryption as agent keys). On startup gnomcp hydrates sessions from disk and reconciles against the chain. A transient query failure never deactivates a session — the chain client returns a sentinel (`ErrSessionQueryUnsupported`) and local state stays authoritative until a definitive answer arrives.

**Observability.** Every write result names the signing identity (`Signed by: agent … / session … on behalf of …`); the audit log records the session address for session-signed writes. State transitions happen inside tool calls — no background polling.

## Alternatives considered

**Master-key signing (v1 model).** Rejected: state-changing tools + untrusted input + sensitive key access is exactly the trifecta to avoid; compromise of the MCP must not equal compromise of the user.

**Sessions as the only write path.** The original v2 position. Superseded: on allowlisted chains (testnet funds only), an MCP-owned agent key gives a zero-ceremony default write path, while sessions remain the path for acting as the user. The session-only posture made every first write a human ceremony for stakes that don't warrant it.

**Fund-to-authorize (the v1 PR #3 model — MCP owns a key, user authorizes by funding it).** Partially adopted: the MCP-owned-key idea became the agent identity, and the actionable `authentication_required`-style error flow survives. The balance-as-authorization semantics were rejected — funding conveys no scope, no identity binding, and no revocation; chain-enforced sessions are categorically stronger for acting as a user.

**In-memory-only session keys.** Rejected as default: per-restart re-ceremony is unacceptable UX. On-disk with file perms + opt-in encryption matches how `gnokey` itself stores keys, and the privkey is bounded by chain-enforced scope regardless of where it lives.

**Per-profile scope without hardcoded clamps.** Rejected: most users never edit `profiles.toml`; ceilings must be structural, with `bypass-hard-limits` as the deliberate escape hatch.

## Consequences

- Master-key exposure is closed structurally; blast radius of a compromised session is capped by chain-enforced `AllowPaths`/`SpendLimit`/`ExpiresAt`, and of a compromised agent key by its testnet balance.
- Writes work out of the box (agent identity) while acting-as-the-user stays an explicit, user-signed grant.
- The master address must be configured per profile (`master-address`); automatic discovery is deferred (see the master-discovery ADR). Dynamic profiles therefore cannot do session writes.
- Open-ended invocation remains: a session covers whole realms, not specific functions; the agent key is unrestricted within its balance. Typed task interfaces (a2a ADR) are the planned narrowing.
- The session path is functional but young — user-facing docs mark it WIP and advise tight `allow_paths`, low `spend_limit`, short `expires_in`. Reworking it (UX and internals) is expected before any stability claim.
