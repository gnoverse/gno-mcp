# Build, test, configure

> **Category: author tooling.** Update when the `gno` binary subcommand surface, `gnomod.toml` schema, or filetest directive surface changes.
> **Authoritative spec**: `docs/resources/gno-testing.md`, `docs/resources/configuring-gno-projects.md`, and `docs/resources/go-gno-compatibility.md` in the `gnolang/gno` repo. This reference is the practical author-oriented summary.

## Why this reference exists

End-user-authors writing realms need to **scaffold projects**, **run tests**, **understand the Go subset**, and **ship to chain**. The mechanics are stable enough to document succinctly here; the upstream docs are the source of truth for full details.

## Project layout

A minimal realm:

```
counter/
├── gnomod.toml
├── counter.gno
└── counter_test.gno
```

A workspace with multiple packages:

```
project/
├── gnowork.toml          # marks the workspace root
├── p/library/
│   ├── gnomod.toml
│   └── lib.gno
└── r/app/
    ├── gnomod.toml
    └── app.gno
```

The `gnowork.toml` lets `gno test` and `gno lint` resolve cross-package imports within the workspace before falling back to the package cache (`$GNOHOME/pkg/mod/`).

## `gnomod.toml`

Generated by `gno mod init <pkgpath>`. Required by Gno tooling. Old `gno.mod` files convert via `gno mod tidy`.

```toml
module = "gno.land/r/example/counter"
gno = "0.9"
```

| Field | Purpose |
|---|---|
| `module` | Canonical package path. Must match the path used in the `addpkg` transaction. |
| `gno` | Gno language version. Currently `"0.9"`. |
| `draft` | Marks the package as unimportable during normal operation. **Ignored at block 0** (genesis-only). Chain-creator flag. |
| `private` | Marks the package as unimportable by any other package. Can be **re-uploaded** (new version overwrites). Memory/pointers/types from a private package cannot be stored in others. Typical use: `r/<user>/home`. Does not provide privacy — all code is still public. |
| `ignore` | Marks the module as ignored by the toolchain but still usable in dev environments. |
| `replace` | (coming soon) Local development; causes `addpkg` to fail on-chain. |
| `[addpkg]` | On-chain metadata (creator address, block height). Filled by the VM keeper at deploy. Not for manual use off-chain. |

## `gnowork.toml`

Empty file at the workspace root. No configuration options yet — its presence is the signal. Resolves dependencies between packages in the workspace before consulting the remote cache.

## The `gno` binary

Subcommands you actually use day-to-day:

| Command | Purpose |
|---|---|
| `gno mod init <pkgpath>` | Scaffold `gnomod.toml` |
| `gno mod tidy` | Convert legacy `gno.mod`; resolve missing deps |
| `gno mod download` | Fetch a dependency without running anything |
| `gno test .` | Run tests in the current package |
| `gno test -v .` | Verbose; list each test |
| `gno test -run "_filetest.gno" .` | Only filetests |
| `gno test --update-golden-tests .` | Update expected filetest outputs |
| `gno run -expr "println(...)"` | Evaluate an expression in package context |
| `gno run -debug -expr "..."` | Start the GnoVM debugger |
| `gno fmt` | Format `.gno` files (gofmt-compatible) |
| `gno lint` | Lint (style + correctness) |
| `gno doc <pkgpath>` | Show package documentation |
| `gno clean -modcache` | Remove `$GNOHOME/pkg/mod/` |

`gno tool transpile` and `gno tool repl` exist for advanced cases — transpile to Go for benchmarking, REPL for ad-hoc evaluation.

## Testing — three flavors

| Flavor | Filename | Use for |
|---|---|---|
| Unit test | `*_test.gno` | Standard `testing.T` style — assert function-level behavior |
| File test | `*_filetest.gno` | Golden test with a directive comment block declaring expected output |
| Integration | external (`gnodev`, txtar) | End-to-end against a real chain; catches `MsgCall` vs `MsgRun` semantic mismatches |

### Unit tests

```go
package counter

import "testing"

func TestIncrement(t *testing.T) {
    if count != 0 {
        t.Fatalf("Expected 0, got %d", count)
    }
    value := Increment(cross(cur), 42)
    if value != 42 {
        t.Fatalf("Expected 42, got %d", value)
    }
}
```

Use the same `testing.T` API as Go. Helpers from the `testing` stdlib package let you fake EOAs, set inbound `OriginSend`, and install fake realms — see `stdlib.md` § `testing`.

### Filetests

Filetests are **golden tests**: a self-contained file with a `main` function and a directive block declaring expected output. Stored in a `filetests/` subdirectory.

> **Stability notice (per upstream)**: filetests are primarily an internal tool. Their API is not guaranteed to be as stable as `gno test`.

```go
// PKGPATH: gno.land/r/demo/counter_test
// SEND: 1000000ugnot
package counter_test

import "gno.land/r/demo/counter"

func main() {
    counter.Increment(cross(cur))
    println(counter.Render(""))
}

// Output:
// 1
```

**Input directives** (top-of-file single-line comments):

| Directive | Description | Default |
|---|---|---|
| `PKGPATH` | Package path. Use `r/` for realms. | `main` |
| `MAXALLOC` | Max memory allocation in bytes. | `0` |
| `SEND` | Coins sent with the transaction. | (none) |

**Output directives** (bottom multi-line comment block):

| Directive | Matches |
|---|---|
| `Output` | Standard output |
| `Error` | Panic or error message |
| `Realm` | Realm state change operations |
| `Events` | Emitted events (JSON) |
| `Preprocessed` | Preprocessed AST |
| `Stacktrace` | Gno stacktrace on panic |
| `Gas` | Gas consumed |
| `Storage` | Realm storage size diff |
| `TypeCheckError` | Go type-checker error |

**Pure package gotcha**: imports of pure packages are processed separately. If a `/p/` import does `println(1)`, its output cannot be checked by an `// Output:` directive on the importing realm.

### Integration via `gnodev`

For cross-realm flows, txtar-style harnesses, and EOA/MsgRun semantic checks, `gnodev` runs an interactive in-memory chain. Use it when designing public surfaces — the txtar log shows what callers see in practice.

### Payment-guard regression test

For realms accepting payment, **always include a test that asserts the payment guard rejects `IsUserRun` callers**. Use `testing.NewCodeRealm()` to simulate an intermediate attacker realm. Without this, the `IsUser()` vs `IsUserCall()` regression is invisible to unit tests that only fake `IsUserCall` callers. See `security.md` § Payment-guard.

## Mocked environment caveats

The `gno` binary mocks a blockchain environment during testing and `gno run`. Consequences:

- **No connection to a real chain.** State changes are in-memory only.
- **Modifying state twice does not stack.** Running the same expression twice resets the in-memory state each time; you may see the same result both times instead of accumulating.
- **Imports resolve from the GnoVM's installation directory**, not from the live chain. To pull a fresh version from the chain, `gno mod download` then re-run.
- **`gnodev` is the closest local approximation** of real chain behavior — use it when the mocked environment lies about something.

## Go-Gno compatibility — what you can and cannot do

Gno is modeled after Go 1.17. Most Go syntax works; the differences matter when you hit them.

### Missing language features

| Feature | Status |
|---|---|
| `go` (goroutines) | missing (after launch) |
| `select` | missing (after launch) |
| `chan T` (channels) | missing (after launch) |
| Generics | not implemented |
| Type shadowing of built-ins | not allowed (`rune := rune('a')` rejected) |

No concurrency primitives, no generics. Plan around this.

### Persistable types

| Type | Persistable? |
|---|---|
| `bool`, integer types, floats, `string`, `rune`, `byte`, `interface{}`/`any` | yes |
| `complex64`, `complex128` | missing |
| `uintptr`, `unsafe.Pointer` | not supported |
| Slices, arrays, maps, pointers, structs, functions | yes (depends on element types) |

Additional Gno builtins:
- `bigint` — based on `math/big.Int`.
- `bigdec` — based on cockroachdb/apd.

**Determinism note**: converting a `string` to `[]byte` or `[]rune` produces a slice with `cap == len`. Code that observes `cap()` after such a conversion gets deterministic results.

### Stdlibs — what works

The Gno stdlib (`chain`, `chain/runtime`, `chain/banker`, `chain/markdown`, `testing`) is the canonical surface. See `stdlib.md`.

Go stdlibs are mostly present but with caveats. Categories of unavailable / restricted Go stdlibs in upstream `docs/resources/go-gno-compatibility.md`:

- **`nondet`** — would require non-determinism (cryptographic randomness, OS/network access). May exist with a different API.
- **`gospec`** — Go-specific (Go AST, debugging info). A Gno equivalent may exist with a different package name.
- **`gnics`** — requires generics; not yet usable.
- **`test`** — available in test contexts only, with limited functionality.
- **`cmd`** — Go stdlib commands; no useful direct equivalent in Gno.

**Will not work**: `net`, `os`, `syscall`, `runtime` (the Go one — `chain/runtime` is the Gno one). The VM has no host access. See `stdlib.md` § What NOT to import.

**Works with caveats**: `time.Now()` is resolved deterministically from block time (usable but plan for determinism).

For the exhaustive table, load `docs/resources/go-gno-compatibility.md` from the gno repo.

## Deploying

Off-chain author tooling stops at the build. Deploy uses `gnokey maketx addpkg` against a chain. Recap:

- The `module` field in `gnomod.toml` must match the path used in the `addpkg` transaction.
- The deployer's address is captured at deploy time via `runtime.OriginCaller()` in `init()`. After init, the deployer is no longer accessible from realm code.
- Storage cost is paid out of the deployer's locked GNOT deposit. Use the `-max-deposit` flag to cap.

For details on deploy mechanics, gnoweb interaction, and chain interaction surfaces, see the upstream `docs/users/` directory.
